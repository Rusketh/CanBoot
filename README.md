# CanBoot

**CanBoot** is the bare-metal bootable runtime for the
[CanDo](https://github.com/Rusketh/CanDo) language. It is one freestanding
ELF — kernel, drivers, networking stack, TLS, and the CanDo VM — that
boots on UEFI and BIOS hardware on x86_64 and aarch64 and drops straight
into a `.cdo` script as its userspace.

It exists so you can write a custom boot tool — a PXE installer, a
recovery shell, a kiosk image, a diagnostic environment — entirely in
CanDo. No Linux, no busybox, no second runtime to learn. Boot the image,
the kernel hands control to `/init.cdo`, you have framebuffer + input +
disks + network + TLS + crypto + media decoding as CanDo namespaces.

### Why CanBoot exists

CanDo is a small, embeddable language with a real threading runtime and
~25 libraries. The natural shape for a CanDo program is *the thing the
machine runs* — not a script inside a host application written in
something else. CanBoot is that host: a thin freestanding kernel whose
only job is to set up enough of the machine (long mode, framebuffer,
PCI, virtio, AHCI, FAT32/ISO9660/ext4/NTFS, lwIP, Mbed TLS) to make
CanDo's stdlib do what it says on the tin, then get out of the way.

The runtime contract is the same in C and in `.cdo`: one kernel, real
hardware, no userspace OS between the script and the device. A
`display.pixel(...)` call lands on the framebuffer BAR; a
`net.connect(...)` call walks lwIP into virtio-net; a
`tls.handshake(...)` call drives Mbed TLS over a cooperative BIO. There
is no abstraction layer outside what CanDo itself exposes.

The architecture and design were planned by humans. **The

## Quick start

### Build from source

```bash
cmake -B build -G Ninja -DCANBOOT_ARCH=x86_64
cmake --build build --target canboot-x86_64
cmake --build build --target canboot-uefi
```

The two targets produce the BIOS kernel ELF (`build/canboot-x86_64.elf`,
loaded by GRUB via Multiboot2) and the UEFI PE/COFF application
(`build/canboot-x86_64-uefi.efi`, loaded by firmware directly).

Wrap either as a bootable image and run it under QEMU:

```bash
bash scripts/mkiso/mkiso-bios.sh build/canboot-x86_64.elf build/canboot-bios.iso
bash tests/run-qemu-bios.sh         build/canboot-bios.iso

bash scripts/mkiso/mkiso-uefi.sh build/canboot-x86_64-uefi.efi build/canboot-uefi.iso
bash tests/run-qemu-uefi.sh         build/canboot-uefi.iso
```

For aarch64, configure with `-DCANBOOT_ARCH=aarch64
-DCMAKE_TOOLCHAIN_FILE=cmake/toolchain-aarch64.cmake` and target
`canboot-aarch64` (direct kernel boot) or `canboot-aarch64-uefi`
(PE/COFF for AAVMF).

Host build tools: `gcc`, `cmake`, `ninja-build`, `meson`, `python3`,
`grub-pc-bin`, `grub-common`, `gnu-efi`, `ovmf`, `xorriso`, `mtools`,
`dosfstools`, `qemu-system-x86`, `gcc-aarch64-linux-gnu`,
`qemu-system-arm`. Full per-platform setup is in
[`docs/building.md`](docs/building.md).

### Hello world

```cdo
// init.cdo — runs as the kernel's userspace.
display.clear(0x000020);
display.text(40, 40, "Hello from CanBoot", 0xFFFFFF);

VAR key = input.waitKey(5000);
print("got key: ", key);
```

Build a FAT32 disk carrying this script with
`scripts/mkdisk/mkdisk-fat32.sh`, attach it as virtio-blk, and the
kernel will find `/init.cdo`, hand control to CanDo, and your script
runs on the bare metal.

## What's in the box

**Boot path** — Multiboot2 (BIOS via GRUB) and UEFI PE/COFF (via
gnu-efi) loaders on x86_64; direct-kernel and AAVMF on aarch64. Both
firmware paths populate the same `boot_info` (framebuffer, memory map,
ACPI RSDP, command line) before dispatching the shared kernel.

**HAL** — console (16550 / PL011), input (PS/2, virtio-input), disk
(virtio-blk, AHCI), display (linear framebuffer with pixel / line /
text / image primitives, virtio-gpu on aarch64), net (virtio-net),
audio (Intel HDA on x86, virtio-snd on aarch64), PCI(e) enumeration,
virtio-pci transport.

**Filesystems** — read-only ISO9660 (boot path), read+write FAT32 (root
directory), read+write ext4 (lwext4) and NTFS (libntfs-3g + mkntfs),
GPT/MBR partition tables.

**Runtime** — picolibc 1.8.11 as the C library, a preemptive-capable
thread scheduler (`rt/sched`) with a full register context switch behind
the POSIX pthread surface, and a heap carved from the largest usable
region in `boot_info->mmap[]` (hundreds of MiB under QEMU; serialised
behind the preempt/SMP allocator guard). Forced timer preemption is on:
the x86_64 LAPIC timer and the aarch64 GICv2 + virtual generic timer both
drive `canboot_sched_on_tick`. x86_64 SMP boots the Application Processors
by default on the BIOS kernel (ACPI MADT discovery + AP trampoline; each
AP reloads the GDT/IDT and repeats per-CPU SSE + FS_BASE setup, then joins
the global run queue) — a selftest confirms work runs on >1 CPU. SMP is
not enabled on the UEFI image: the trampoline's absolute (`movabs`)
relocations are incompatible with the gnu-efi PIC link, so that build runs
uniprocessor. See `rt/sched/include/sched/sched.h` and
`arch/x86_64/smp.h`.

**Network + TLS** — lwIP 2.2.1 in NO_SYS mode over virtio-net (DHCP /
UDP / TCP / HTTP); Mbed TLS 3.6.x LTS with hardware entropy
(RDSEED/RDRAND + TSC jitter fallback), session tickets, and a
cooperative BIO over lwIP's raw API.

**Media** — stb_image (PNG/JPG/BMP decode), minimp3 (MP3 decode), a
LOVE-2D-shaped audio mixer with multiple concurrent Sources.

**CanDo bindings** — the cando stdlib's bare-metal namespaces:
`display`, `input`, `time`, `file`, `fs`, `disk`, `partition`, `pci`,
`fb`, `env`, `log`, `fmt`, `net`, `http`, `https`, `tls`, `crypto`,
`random`, `encoding` (hex, base64, url), `image`, `audio`. CanDo
scripts get one runtime, real hardware, no userspace shim.

## Repository layout

```
arch/<arch>/             architecture entry, mode transition, IDT/MMU
boot/multiboot2/         Multiboot2 header for BIOS GRUB
boot/uefi/               PE/COFF EFI loader (gnu-efi)
kernel/                  unified kmain, boot_info schema, framebuffer
hal/                     hardware abstraction layer (console, input,
                         disk, display, net, audio, pci, virtio)
fs/                      ISO9660, FAT32, ext4, NTFS, partition tables
net/                     lwIP + Mbed TLS ports
rt/                      picolibc syscall stubs, thread scheduler (sched),
                         spinlocks (sync), POSIX pthread shim
cando_port/
  lib/                   CanDo language bindings (one per namespace)
  runtime/               POSIX function stubs
  jit/                   per-arch JIT codegen stubs
  vendor_glue/           lwext4 / ntfs-3g / minimp3 / stb glue
  shims/                 POSIX header shims for bare-metal builds
vendor/                  submodules: cando, picolibc, lwip, mbedtls,
                         gnu-efi, ntfs-3g, lwext4, stb, minimp3
scripts/mkiso/           ISO image builders (BIOS, UEFI)
scripts/mkdisk/          disk image builders (FAT32, ext4, NTFS, …)
tests/                   QEMU smoke tests + sidecars
tests/selftest/          in-kernel selftests run during boot
include/canboot.h        single public header for external consumers
linker/                  per-arch linker scripts
docs/                    documentation
```

## Status

CanBoot boots to a `/init.cdo` prompt on both firmware paths on x86_64
and on aarch64. The HAL covers serial, input, disk (virtio-blk + AHCI),
framebuffer (with pixel readback assertions in CI), virtio-gpu on
aarch64, virtio-net, and audio (Intel HDA + virtio-snd). Filesystems
cover ISO9660 / FAT32 / ext4 / NTFS read+write including mkfs. The
network stack runs DHCP, TCP/UDP, HTTP, and TLS 1.2 with session
resumption against pinned CAs. The CanDo VM runs unmodified with ~25
bare-metal bindings registered. CI matrices BIOS, UEFI x86_64, and
both aarch64 paths to green on every push.

Chronology lives in `git log` and on the
[Releases page](https://github.com/Rusketh/CanBoot/releases).
